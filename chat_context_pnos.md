## 🎯 СУТЬ И ЦЕЛЕВАЯ АУДИТОРИЯ
Десктопное приложение на фреймворке **PySide6 (Qt)** для автоматизации создания проектных отчетов (ПНОС). Программа парсит файлы Excel и Word, скачивает данные из облака (Яндекс.Диск) и управляет скрытыми процессами Microsoft Office (`win32com.client`) для сложной склейки (Mail Merge) таблиц и форм.

---

## 🏗 АРХИТЕКТУРА И ФАЙЛОВАЯ СТРУКТУРА

Программа разделена на Frontend (UI на главном потоке) и Backend (набор изолированных `QThread` воркеров для каждого логического этапа, чтобы интерфейс не "висел").

### 📦 Зависимости (pip)
`PySide6`, `pywin32` (для `win32com`), `aiohttp` и `aiofiles` (для работы с сетью), `python-docx` (для парсинга Word без COM), `Pillow` (для расчетов размеров фото), `rapidfuzz` (для нечеткого поиска папок).

### 1. Ядро и Интерфейс
- **`main_window.py`**: Главный класс приложения. 
  - Реализует боковое меню с вкладками этапов и экраном "Настройки". 
  - Отвечает за маршрутизацию логов (каждый воркер шлет сигналы `log.emit` и `progress.emit`, которые отрисовываются здесь в красивых логах под прогресс-барами).
  - Хранит логику чекбокса **"Автоматический режим"** (переключение между этапами без участия пользователя).
  - Хранит функцию **Экстренной остановки** (убивает запущенные процессы `EXCEL.EXE` и `WINWORD.EXE`, если юзер нажал "Отмена").
- **`config.py`**: Глобальный словарь `CONFIG`. Содержит API-токены Яндекса (`TOKEN_ITEMS`, `TOKEN_BASE`), список целевых путей скачивания `DISK_PATHS_ITEMS`, имена файлов шаблонов `TEMPLATE_FILES` (`ПНОС_ТТП.docx` и `ПНОС_ТТП.xlsm`), пороги нечеткого поиска (`FUZZY_THRESHOLD`) и путь к сводному мастер-файлу `MACRO_MASTER_WB_PATH`.

### 2. Взаимодействие с Сетью
- **`yandex_api.py`**: Класс `YandexDiskAPI`. Обёртка над REST API Яндекс.Диска.
  - **Важно:** Совершенно недавно этот модуль был полностью переписан на асинхронный движок! Использует `aiohttp.ClientSession` и управляет лимитами через `asyncio.Semaphore(50)`, чтобы не получать 429 ошибку (Too Many Requests). Качает файлы напрямую на диск через `aiofiles`.

### 3. Backend Воркеры (`workers/`)

Каждый этап представляет собой наследника `QThread`. Изолированно читает и пишет данные в папки-пункты, расположенные на рабочем столе юзера.

#### 📁 Этап 1: Скачивание (Сборка структур) — `workers/stage1_worker.py`
- *Суть*: Читает номера пунктов из Реестра или GUI. Ищет папку с этим пунктом на Яндекс.Диске среди путей из списка `DISK_PATHS_ITEMS`. Скачивает только нужные внутренние папки ("Паспорт", "Первичка") и подкидывает в каждую папку локальные файлы-шаблоны `ПНОС_ТТП`.
- *Техника*: Внутри метода `run()` поднимает собственный Event Loop (`asyncio.run(self.async_run())`), после чего сотни сетевых вызовов `YandexDiskAPI` летят параллельно через `asyncio.gather()`. 
- *Фишка*: Умеет дозагружать пункты "на лету" (если пользователь добавил новые пункты в интерфейс, он скачает только их, пропустив готовые).

#### 📝 Этап 2: Копирование Актов (Без Word COM) — `workers/stage2_worker.py`
- *Суть*: Заходит в каждую локальную папку "Первичка", ищет там "Акт замеров толщины стенки.docx" и достает оттуда Таблицу №3.
- *Техника*: **Использует `python-docx`**! Мы жестко вырезали код `win32com.client.Dispatch("Word.Application")` для чтения файла, так как это работало невероятно медленно. Теперь `doc = docx.Document(path)`, берем `doc.tables[2]`, считываем текст в массив списков и затем через Excel COM отдаем его локальному файлу `ПНОС_ТТП.xlsm` на лист "Импорт". Скорость этапа — доли секунды.

#### 🧮 Этап 3: Подготовка расчетов (Мастер-файл & F9) — `workers/stage3_worker.py`
- *Суть*: Заполняет `ПНОС_ТТП.xlsm` и подготавливает сводные таблицы к слиянию.
- *Сложная логика кросс-референса (Свежая доработка)*: 
  Перед тем, как перебирать папки с пунктами, скрипт невидимо открывает "Главный сводный график" (файл по пути из `CONFIG["MACRO_MASTER_WB_PATH"]`). 
  Он переходит на лист **"ПЛАН"** и считывает весь диапазон от `A1` до `B<последняя_строка>` в память (в Python-словарь). 
  Затем скрипт берет номер локальной папки (например, 2758 из `п.2758`), заходит в этот сохраненный словарь (где ключ — Столбец B), достает соответствующий ID (из Столбца А) и именно его вписывает в ячейку **F9** на листе "паспорт" локального `ПНОС_ТТП.xlsm`.
- Обновляет Pivot Tables (`RefreshAll()`).

#### 🔀 Этап 4: Слияние (Mail Merge) — `workers/stage4_worker.py`
- *Суть*: Генерация Word-отчета на базе данных из Excel.
- *Техника (Самое слабое место программы)*:
  1. Через Excel COM запускает внутренний VBA-макрос `modTransTab.TransTab` внутри файла пункта. 
  2. Этот макрос генерирует лист "Экспорт", который скрипт копирует в чистый файл `CleanSource.xlsx`.
  3. Через Word COM открывает `ПНОС_ТТП.docx` и запускает MailMerge, где источником выступает `SELECT * FROM [экспорт$]`.
  4. Замораживает поля и сохраняет финальный `Отчет_XXXX.docx`.
- *DialogKiller*: Для Word параллельно работает поток `WordDialogKiller`, который через Win32GUI API постоянно ищет всплывающие диалоги SQL-подтверждения и жмет на них Enter. Без этого COM-процесс намертво зависнет.

#### 📸 Этапы 5 и 6: Обработка и вставка фотографий (PIL) — `workers/stage5_worker.py`
- *Суть*: Вставка фото и эскизов из папки "Первичка" в конце отчета. Включает сложный алгоритм распознавания цветов и позиционирования.
- *Алгоритм Цвета*: Оценивает преобладающие цвета пикселей фото: бледные синие схемы (>1.15) классифицируются как "Формуляр 02", остальные — "Формуляр 01" (Красные фото).
- *Алгоритм Вставки (Свежая доработка)*: 
  Программа ищет теги `[ФОТО1]` и `[ФОТО2]`. Файлы сортируются через Regex (`_1.png`, `_2.png`). Страницы с тегами клонируются под количество фотографий. Самое важное: программа больше не полагается на кривой внутренний механизм Word `LockAspectRatio`. Сначала изображение скармливается модулю **`PIL`** (Pillow), вычисляются его точные габариты. Если фото больше рамки `520x750`, скрипт математически делит обе стороны на коэффициент масштабирования. Таким образом картинка **абсолютно без искажений** вставляется в Word по центру страницы.

## 🚀 ЧТО ДЕЛАТЬ ДАЛЬШЕ (Текущие задачи для нового ИИ)

Программа уже вылизана и очень функциональна, но есть две очевидные задачи на перспективу, о которых просил юзер:

1. **Рефакторинг Этапа 4 (Отказ от Word COM)**: 
   Текущая реализация (COM MailMerge + WordDialogKiller + Excel Macro) работает медленно и рискует "зависнуть" при малейшем сбое Windows. 
   **Ваша задача**: Предложить решение на замену. Например, использовать **`docxtpl`** (DocxTemplate). Взять `ПНОС_ТТП.docx`, конвертировать его в Jinja2-шаблон (заменить теги на `{{ var_name }}`), вытащить данные из листа "Экспорт" с помощью `openpyxl` или `pandas` и за миллисекунды генерировать финальный отчет, не запуская `WINWORD.EXE` вообще!

2. **Доработка интерфейса "Настройки"**: 
   Сейчас часть критических переменных зашита в `config.py` (например, огромный список `DISK_PATHS_ITEMS` — пути поиска папок на Я.Диске).
   **Ваша задача**: Разработать в `main_window.py` GUI-виджеты (например, `QListWidget` для добавления/удаления путей Я.Диска, ползунки для настройки скорости `asyncio` семафора и настройки цветочувствительности для стадии фото). Сохранять эти настройки в отдельный JSON файл.
